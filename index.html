Why

The wrong way to train is the default way

The problem

Models learn from code. Code teaches them patterns that are as often wrong as right. A code snippet is as likely to be a bug as a feature. Training on code creates conflicts — the model internalizes bad practices alongside good ones. Every "fix" carries the risk of repeating what it learned from broken code.

The fix: Adversarial Alignment

Steer the model with real-world use cases. Human workflows. Actual decisions. Genuine tradeoffs. The model learns what matters from what humans actually do — not from what compiled. CONTRACTX gives it guard rails. The persona matrix gives it checks and balances. King has context + contracts. Workers have fresh windows + no bias. God ensures direction stays true.

The edge

Code-trained models drift. They remember broken patterns from broken repos. Real-world steered models stay aligned — because their reference point is human behavior, not code behavior. Adversarial Alignment is the principle. Kingdom of God is the implementation.

The principle

Adversarial Alignment

Steer, don't code-train

Adversarial Alignment means the model learns from scenarios, not from code. Give it a real workflow: "Here's what the user needed. Here's what we deployed. Here's why we chose this over that. Here's what broke. Here's how we fixed it." The model learns principles — not syntax patterns.

Why adversarial? Because you're actively working against the model's default behavior. Left alone, it pattern-matches against code it's seen. Adversarial Alignment redirects it — "forget the code pattern. Here's what actually happened." The persona matrix makes this systematic: God defines the scenario. King contextualizes it with contracts. Workers execute with fresh eyes.

Personas

The three roles

Every task in the kingdom flows through three personas. Each has a distinct edge.

God — Supreme Leader

Sets direction. Approves major decisions. The final authority on what ships. God doesn't execute — God decides. The kingdom exists to serve God's intent.

Edge: Ultimate context. Knows what matters and what doesn't. No model can replace this — it's human judgment applied at the architectural level.

King — Orchestrator + Contract Master

Walks the CONTRACTX chain. Reads every contract before touching any file. Has the extra edge of accumulated context — knows what was tried, what failed, what the rules say.

Edge: Contract-bound intelligence. King doesn't guess — contracts, not memory. If King is smart enough, handles Worker requests directly without God relaying. God steps in only for major decisions.

Workers — Clean Context Specialists

Every Worker starts with a fresh context window. No bias. No "we tried that last session." No baggage. Reads CONTRACTX before starting — contract-bound, not memory-bound.

Edge: Unbiased execution. While King carries the weight of accumulated context, Workers bring fresh eyes. They see what King might miss. They execute without the assumptions King has internalized.

Memory

LOS — Long-term memory layer

Optional but powerful. While the three operational skills handle each session in isolation, LOS carries knowledge across sessions, projects, and time. Think of it as the kingdom's wiki — available to every persona, governed by policies.

Retention policy

What stays and what expires. Knowledge files persist forever. Daily logs accumulate but can be pruned. Tasks archive when done. The retention policy means LOS doesn't become a junkyard — it stays a clean, searchable memory.

Policy: identity.md (permanent) → knowledge/ (permanent) → logs/ (retention window) → .tasks/ (archive on completion).

Ingest policy

What gets consumed and how. King reads LOS for context before routing. Workers don't — they get fresh windows. Agent Retro writes session learnings back to LOS. Shit Hit The Fan reads LOS for incident history. CONTRACTX writes gap resolutions to LOS.

Policy: King reads, retro workers write, operational workers don't — they stay clean.

Cross-project memory

Drop a .claude.los file in any project root and it links to your LOS. King reads your context — who you are, what you know, what you decided — across every project. No repeated explanations. No "what did we decide last week?"

Policy: one LOS repo, many projects linked. Memory follows you everywhere.

FAQ

Questions about the kingdom

Why not just code everything?

Code teaches models wrong patterns. A code snippet is as often a bug as a feature. Training on code creates conflicts — the model learns both good and bad practices and can't tell them apart. Adversarial Alignment steers the model with real-world scenarios instead. Human behavior, not code behavior.

What's the edge King has that Workers don't?

King has accumulated context — knows contractual history, what was tried, what failed, what the rules say. King walks the CONTRACTX chain before every action. Workers have the opposite edge: fresh context windows with no bias from previous sessions. King routes. Workers execute with clean eyes.

When does God step in vs when does King handle things?

Major decisions go through God: new features, deploys, architecture changes, direction-setting. King handles routine routing autonomously. If King is smart enough to handle Worker requests without God relaying, it does. God is the supreme leader — not a bottleneck.

Why do Workers need CONTRACTX?

Workers start with fresh context. Without contracts, they have no guard rails — no knowledge of conventions, invariants, or boundaries. CONTRACTX gives them rules without giving them baggage. Contract-bound, not memory-bound.

What is Adversarial Alignment exactly?

Instead of training models on code (which teaches them bugs and bad patterns), you steer them with real human use cases. You're adversarial because you're working against the model's default behavior — its pattern-matching against code. You redirect it: "forget the code. Here's what happened in reality." The kingdom makes this systematic.
